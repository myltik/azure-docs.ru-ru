# <a name="run-a-disaster-recovery-drill-for-hyper-v-vms-to-your-secondary-on-premises-site"></a>Проверка отработки аварийного восстановления на вторичный локальный сайт для виртуальных машин Hyper-V

Служба [Azure Site Recovery](site-recovery-overview.md) помогает реализовать стратегию аварийного восстановления за счет управления процессами репликации, отработки отказа и восстановления локальных компьютеров и виртуальных машин Azure.

Из этого учебника вы узнаете, как проверить отработку аварийного восстановления на вторичный локальный сайт для виртуальных машин Hyper-V. Управление виртуальными машинами Hyper-V должно осуществляться в частном облаке System Center Virtual Machine Manager (VMM). Проверка отработки позволяет проверить стратегию репликации без потери данных и простоя и не влияет на рабочую среду. Из этого руководства вы узнаете, как выполнять такие задачи:

> [!div class="checklist"]
> * Выполнение предварительных требований для тестовой отработки отказа
> * Запуск тестовой отработки отказа для одного компьютера


## <a name="prerequisites"></a>Предварительные требования

- Тестовую отработку отказа можно запустить для нескольких вариантов сети на вторичном сайте. Ее можно запустить без подключения к сети, в существующей сети или автоматически создав тестовую сеть с помощью Site Recovery. 
**Если вы хотите использовать существующую рабочую сеть для тестовой отработки отказа, обратите внимание на следующее**:
    - При выполнении тестовой отработки отказа основную виртуальную машину следует выключать. В противном случае в одной сети будут одновременно работать две виртуальные машины с одинаковыми идентификаторами. 
    - Если тестовые виртуальные машины были изменены, то при очистке тестовой отработки отказа эти изменения будут утеряны. Изменения не реплицируются обратно на основную виртуальную машину.
    - Тестирование рабочей сети приводит к простою рабочих нагрузок. Попросите пользователей не запускать соответствующие приложения во время проверки аварийного восстановления. 
- Тип тестовой сети реплики не обязательно должен соответствовать типу логической сети VMM, используемой для тестовой отработки отказа. Однако некоторые сочетания типов не будут работать. Например, если в реплике используется DHCP и изоляция на основе виртуальной локальной сети, вы не можете использовать виртуализацию сети Windows для тестовой сети отработки отказа, так как для нее требуются пулы IP-адресов. 
- Для тестовой отработки отказа не рекомендуется использовать сеть, выбранную для сетевого сопоставления.


## <a name="run-a-test-failover-for-a-vm"></a>Выполнение тестовой отработки отказа для виртуальной машины

1. В разделе **Реплицированные элементы** выберите виртуальную машину и щелкните **Тестовая отработка отказа**.
2. В области **Тестовая отработка отказа** укажите, как тестовые виртуальные машины будут подключаться к сетям после тестовой отработки отказа. Для выполнения действий, описанных в этом учебнике, рекомендуется автоматически создать тестовую сеть с помощью Site Recovery. [Подробнее](site-recovery-test-failover-vmm-to-vmm.md#network-options-in-site-recovery).
3. Нажмите кнопку **ОК** , чтобы запустить отработку отказа. Отслеживайте ход выполнения на вкладке **Задания** .
4. После отработки отказа проверьте, успешно ли запущены тестовые виртуальные машины.
5. Когда закончите, нажмите кнопку **Очистить тестовую отработку отказа**. При этом будут удалены тестовые виртуальные машины и все сети, созданные во время тестовой отработки отказа.
6. В разделе **Примечания** можно записать и сохранить любые замечания, связанные с тестовой отработкой отказа.
