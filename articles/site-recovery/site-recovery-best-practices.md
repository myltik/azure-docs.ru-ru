# <a name="get-ready-to-deploy-azure-site-recovery"></a>Подготовка к развертыванию Azure Site Recovery

В этой статье описывается подготовка к развертыванию Azure Site Recovery.

## <a name="protecting-hyper-v-virtual-machines"></a>Защита виртуальных машин Hyper-V

Есть несколько вариантов развертывания для защиты виртуальных машин Hyper-V. Вы можете реплицировать локальные виртуальные машины в Azure или вспомогательный центр обработки данных. Требования для каждого варианта развертывания отличаются.

**Требование** | **Репликация в Azure (с помощью VMM)** | **Репликация виртуальных машин Hyper-V в Azure (без VMM)** | **Репликация виртуальных машин Hyper-V на дополнительный сайт (без VMM)** | **Дополнительные сведения**
---|---|---|---|---
**VMM** | Группа VMM, запущенная на System Center 2012 R2 <br/><br/>Минимум одно облако VMM, содержащее одну или несколько групп узлов VMM. | Нет данных | Серверы VMM на первичных и вторичных сайтах, запущенные минимум на System Center 2012 с пакетом обновления 1 (SP1) с последними обновлениями. <br/><br/> Минимум одно облако на каждом сервере VMM. Для всех облаков необходимо настроить профиль емкости Hyper-V.<br/><br/> Исходное облако должно включать минимум одну группу узлов VMM. | необязательный параметр. Вам не обязательно иметь развернутую структуру System Center VMM для репликации виртуальных машин Hyper-V в Azure. В противном случае убедитесь в правильности настройки сервера VMM. Проверьте, правильная ли версия VMM используется и настроены ли облака.
**Hyper-V** | Минимум один сервер узла Hyper-V на локальном сайте под управлением Windows Server 2012 R2 или более поздней версии. | Минимум один сервер Hyper-V в исходном и целевом узлах под управлением Windows Server 2012 с установленными последними обновлениями и подключением к Интернету.<br/><br/> Серверы Hyper-V должны входить в группу узлов в облаке VMM. | Минимум один сервер Hyper-V в исходном и целевом узлах под управлением Windows Server 2012 с установленными последними обновлениями и подключением к Интернету.<br/><br/> Серверы Hyper-V должны входить в группу узлов в облаке VMM. |
**Виртуальные машины** | Минимум одна виртуальная машина на сервере исходного узла Hyper-V. | Минимум одна виртуальная машина на сервере узла Hyper-V в исходном облаке VMM. | Минимум одна виртуальная машина на сервере узла Hyper-V в исходном облаке VMM. |  Виртуальные машины, реплицирующиеся в Azure, должны соответствовать требованиям к виртуальным машинам Azure.
**Учетная запись Azure** | Вам потребуется учетная запись Azure и подписка на службу Site Recovery. | Вам потребуется учетная запись Azure и подписка на службу Site Recovery. | Нет данных | Если у вас нет учетной записи, начните с получения бесплатной пробной версии.
**Служба хранилища Azure** | Вам потребуется учетная запись хранения Azure с включенной георепликацией. | Вам потребуется учетная запись хранения Azure с включенной георепликацией. | Нет данных | Учетная запись должна находиться в том же регионе, что и хранилище Azure Site Recovery, и быть связана с той же подпиской.
**Сеть** | Настройте сетевое сопоставление, чтобы все виртуальные машины, для которых отработка отказа выполняется в одной и той же сети, могли подключаться друг к другу независимо от того, к какому плану восстановления они относятся. Кроме того, если в целевой сети Azure настроен сетевой шлюз, виртуальные машины могут подключаться к другим локальным виртуальным машинам. Если вы не настроите сетевое сопоставление, подключение будет возможно только между машинами, для которых отработка отказа выполняется по одному и тому же плану восстановления. | Нет данных |  <br/><br/>Настройте сетевое сопоставление, чтобы после отработки отказа виртуальные машины могли подключаться к соответствующим сетям, а также чтобы обеспечить оптимальное размещение реплик виртуальных машин на серверах узла Hyper-V. Если не настроить сетевое сопоставление, реплицированные машины не будут подключаться к сетям виртуальных машин после отработки отказа. |  Чтобы настроить сетевое сопоставление с VMM, убедитесь, что логические сети и сети виртуальных машин VMM настроены правильно.
**Поставщики и агенты** | Во время развертывания установите на серверах VMM поставщик Azure Site Recovery. Установите агент служб восстановления Azure на серверах Hyper-V в облаках VMM. | Во время развертывания установите поставщик Azure Site Recovery и агент служб восстановления Azure на сервере узла Hyper-V или в кластере.| Во время развертывания установите на серверах VMM поставщик Azure Site Recovery. Установите агент служб восстановления Azure на серверах Hyper-V в облаках VMM. | Поставщики и агенты подключаются к службе Site Recovery через Интернет с помощью зашифрованного HTTPS-подключения. Для подключения поставщика не требуется добавлять исключения брандмауэра или создавать особый прокси-сервер.
**Подключение к Интернету** | Подключение к Интернету требуется только для серверов VMM. | Подключение к Интернету требуется только для серверов узла Hyper-V. | Подключение к Интернету требуется только для серверов VMM. | На виртуальные машины не нужно ничего устанавливать. Они не подключаются к Интернету напрямую.



## <a name="protect-vmware-virtual-machines-or-physical-servers"></a>Защита виртуальных машин VMware или физических серверов

Есть несколько вариантов развертывания для защиты виртуальных машин VMware или физических серверов Windows и Linux. Вы можете реплицировать эти объекты в Azure или дополнительный центр обработки данных. Требования для каждого варианта развертывания отличаются.

**Требование** | **Репликация виртуальных машин VMware или физических серверов в Azure** | * **Репликация виртуальных машин или физических серверов VMware на дополнительный сайт**  
---|---|---
**Основной сайт** | **Сервер обработки**: выделенный сервер Windows (физический или виртуальный). | **Сервер обработки**: выделенный сервер Windows (физический или виртуальная машина VMware).<br/><br/>  
**Дополнительный локальный сайт** | Нет данных | **Сервер конфигурации**: выделенный сервер Windows (физический или виртуальный). <br/><br/> **Главный целевой сервер**: выделенный сервер (физический или виртуальный). Настраивается с помощью средств Windows или Linux для защиты компьютеров под управлением Windows или Linux соответственно.
**Таблицы Azure** | **Подписка**: потребуется подписка на службу Site Recovery. <br/><br/> **Учетная запись хранения**: вам понадобится учетная запись хранения с включенной георепликацией. Учетная запись должна находиться в том же регионе, что и хранилище Site Recovery, и быть связана с той же подпиской. <br/><br/> **Сервер конфигурации**: потребуется настроить сервер конфигурации в качестве виртуальной машины Azure. <br/><br/> **Главный целевой сервер**: потребуется настроить главный целевой сервер в качестве виртуальной машины Azure. <br/><br/> Настраивается с помощью средств Windows или Linux для защиты компьютеров под управлением Windows или Linux соответственно.<br/><br/> **Виртуальная сеть Azure**: вам понадобится виртуальная сеть Azure, в которой будут развернуты сервер конфигурации и главный целевой сервер. ## <a name="azure-virtual-machine-requirements"></a>Требования к виртуальной машине Azure

Вы можете развернуть Site Recovery для репликации виртуальных машин и физических серверов под управлением любой операционной системы, поддерживаемой Azure. Поддерживаются большинство версий Windows и Linux. Вам нужно убедиться, что локальные виртуальные машины, которые вы хотите защитить, соответствуют требованиям Azure.


## <a name="optimizing-your-deployment"></a>Оптимизация развертывания

Следующие советы помогут вам оптимизировать и масштабировать развертывание.

- **Размер тома операционной системы**. При репликации виртуальной машины в Azure размер тома операционной системы не должен превышать 1 ТБ. Если размер томов превышает это значения, вы можете вручную перенести их на другой диск перед началом развертывания.
- **Размер диска с данными**. При репликации в Azure вы можете разместить на виртуальной машине до 32 дисков с данными по 1 ТБ каждый. Вы можете успешно выполнить репликацию и отработать отказ для виртуальной машины объемом около&32; ТБ.
- **Ограничения плана восстановления**. Служба Site Recovery может включать тысячи виртуальных машин. Планы восстановления предназначены для приложений, отработка отказа которых должна проводиться одновременно. По этой причине максимальное число машин в плане восстановления — 50.
- **Ограничения служб Azure**. Каждая подписка Azure включает набор ограничений по умолчанию для ядер, облачных служб и т. д. Рекомендуем выполнить тестовую отработку отказа, чтобы проверить доступность ресурсов в вашей подписке. Вы можете изменить эти ограничения через службу поддержки Azure.
- **Планирование загрузки**: выделите ресурсы в соответствии с потребностями производительности и масштабирования.
- **Пропускная способность репликации**. При невысокой пропускной способности репликации обратите внимание на следующее.
    - **ExpressRoute**. Служба Site Recovery работает с Azure ExpressRoute и оптимизаторами глобальной сети, такими как Riverbed.
    - **Трафик репликации**. Служба Site Recovery выполняет интеллектуальную начальную репликацию, используя только блоки данных, а не весь виртуальный жесткий диск. Текущая репликация применяется только к изменениям.
    - **Сетевой трафик**. Вы можете контролировать сетевой трафик, используемый для репликации, путем настройки качества обслуживания Windows с применением политики на основе IP-адреса и порта назначения.  Кроме того, при репликации в Azure Site Recovery с использованием агента службы архивации Azure вы можете настроить регулирование для этого агента.
- **Целевое время восстановления (RTO)**. Если вам требуется измерить целевое время восстановления, которое можно ожидать от Site Recovery, рекомендуем выполнить тестовую отработку отказа и просмотреть задания Site Recovery, чтобы определить время, необходимое для выполнения операций. Чтобы достичь наилучшего целевого время восстановления при отработке отказа с перемещением в Azure, рекомендуем автоматизировать все ручные действия путем интеграции со службой автоматизации Azure и планами восстановления.
- **Целевая точка восстановления (RPO)**. Служба Site Recovery поддерживает практически синхронную целевую точку восстановления при репликации в Azure. Для этого необходима достаточная пропускная способность между вашим центром обработки данных и Azure.
