# <a name="troubleshooting-azure-point-to-site-connection-problems"></a>Устранение неполадок подключения типа "точка — сеть" Azure

В этой статье перечисляются распространенные проблемы подключения типа "точка — сеть", которые могут возникнуть. Кроме того, здесь рассматриваются возможные причины этих проблем и способы их устранения.

## <a name="vpn-client-error-a-certificate-could-not-be-found"></a>Ошибка VPN-клиента: не удалось найти сертификат

### <a name="symptom"></a>Симптом

При попытке подключения к виртуальной сети Microsoft Azure с помощью VPN-клиента появляется следующее сообщение об ошибке:

**Не удалось найти сертификат, который был бы использован с протоколом расширенной проверки подлинности (EAP). (Ошибка 798.)**

### <a name="cause"></a>Причина:

Эта проблема возникает, если сертификат клиента отсутствует в папке **Certificates - Current User\Personal\Certificates**.

### <a name="solution"></a>Решение

Убедитесь, что сертификат клиента установлен в следующем расположении в хранилище сертификатов (Certmgr.msc):
 
**Certificates - Current User\Personal\Certificates**

Дополнительные сведения о том, как установить сертификат клиента, см. в разделе [Создание и экспорт сертификатов для подключений типа "точка — сеть" с помощью PowerShell](vpn-gateway-certificates-point-to-site.md).

> [!NOTE]
> При импорте сертификата клиента не выбирайте параметр **Включить усиленную защиту закрытого ключа**.

## <a name="vpn-client-error-the-message-received-was-unexpected-or-badly-formatted"></a>Ошибка VPN-клиента: получено непредвиденное или неправильно отформатированное сообщение

### <a name="symptom"></a>Симптом

При попытке подключения к виртуальной сети с помощью VPN-клиента появляется следующее сообщение об ошибке:

**Ошибка VPN-клиента: получено непредвиденное сообщение или оно имеет неправильный формат. (Ошибка 0x80090326.)**

### <a name="cause"></a>Причина:

Эта проблема возникает, если открытый ключ корневого сертификата не был передан на VPN-шлюз Microsoft Azure либо он поврежден или истек его срок действия.

### <a name="solution"></a>Решение

Чтобы устранить эту проблему, проверьте состояние корневого сертификата на портале Azure, чтобы узнать, не отозван ли он. Если он не был отозван, попробуйте удалить корневой сертификат и повторить его передачу. Дополнительные сведения см. в разделе [Создание сертификатов](vpn-gateway-howto-point-to-site-classic-azure-portal.md#generatecerts).

## <a name="vpn-client-error-a-certificate-chain-processed-but-terminated"></a>Ошибка VPN-клиента: цепочка сертификатов обработана, но обработка прервана 

### <a name="symptom"></a>Симптом 

При попытке подключения к виртуальной сети Azure с помощью VPN-клиента появляется следующее сообщение об ошибке:

**Цепочка сертификатов обработана, но обработка прервана на корневом сертификате, у которого отсутствует отношение доверия с поставщиком доверия.**

### <a name="solution"></a>Решение

1. Убедитесь, что перечисленные ниже сертификаты находятся в правильном расположении.

    | Сертификат | Расположение |
    | ------------- | ------------- |
    | AzureClient.pfx  | Current User\Personal\Certificates |
    | Azuregateway-*GUID*.cloudapp.net  | Current User\Trusted Root Certification Authorities|
    | AzureGateway-*GUID*.cloudapp.net, AzureRoot.cer    | Local Computer\Trusted Root Certification Authorities|

2. Если сертификаты уже находятся в соответствующих расположениях, попробуйте удалить эти сертификаты и установить их заново. Сертификат **azuregateway-*GUID*.cloudapp.net** можно найти в пакете конфигурации VPN-клиента, скачиваемого с портала Azure. Для извлечения файлов из пакета можно использовать архиваторы.

##  <a name="error-file-download-error-target-uri-is-not-specified"></a>Ошибка: "Произошла ошибка скачивания файла. Целевой URI не указан."

### <a name="symptom"></a>Симптом

Отображается следующее сообщение об ошибке:

**Произошла ошибка скачивания файла. Целевой URI не указан.**

### <a name="cause"></a>Причина: 

Эта проблема возникает из-за неправильного типа шлюза. 

### <a name="solution"></a>Решение

Типом VPN-шлюза должен быть **VPN**, а типом VPN — **routebased**.

## <a name="vpn-client-error-azure-vpn-custom-script-failed-8007026f"></a>Ошибка VPN-клиента: сбой настраиваемого сценария VPN Azure (8007026f)

### <a name="symptom"></a>Симптом

При попытке подключения к виртуальной сети Azure с помощью VPN-клиента появляется следующее сообщение об ошибке:

**Custom script (to update your routing table) failed (8007026f)** (Произошел сбой настраиваемого сценария (для обновления таблицы маршрутизации) (8007026f)).

### <a name="cause"></a>Причина:

Это может произойти, если вы пытаетесь открыть VPN-подключение типа "сеть — точка" с помощью ярлыка.

### <a name="solution"></a>Решение 

Откройте пакет VPN напрямую, а не с помощью ярлыка.

## <a name="cannot-install-the-vpn-client"></a>Не удается установить VPN-клиент

### <a name="cause"></a>Причина: 

Для установления отношений доверия между VPN-шлюзом и виртуальной сетью требуется дополнительный сертификат. Этот сертификат включен в пакет конфигурации VPN-клиента, который создается на портале Azure.

### <a name="solution"></a>Решение

Извлеките содержимое пакета конфигурации VPN-клиента. Вы найдете CER-файл. Установите сертификат в папку **Trusted Root Certification Authorities** **учетной записи компьютера**.

1. Запустите mmc.exe.
2. Добавьте оснастку **Сертификаты**.
3. Выберите учетную запись **Компьютер** для локального компьютера.
4. Щелкните правой кнопкой мыши узел **Trusted Root Certification Authoritie** > **Все задачи** > **Импорта** и найдите CER-файл, извлеченный из пакета конфигурации VPN-клиента.
5. Перезагрузите компьютер. 
6. Попробуйте установить VPN-клиент.

## <a name="azure-portal-error-failed-to-save-vpn-gateway-data-is-invalid"></a>Ошибка портала Azure: не удалось сохранить VPN-шлюз, так как данные являются недопустимыми

## <a name="azure-portal-error-failed-to-save-vpn-gateway-resource-name-is-invalid"></a>Ошибка портала Azure: не удалось сохранить VPN-шлюз, так как имя ресурса является недопустимым

### <a name="symptom"></a>Симптом

При попытке сохранить изменения для VPN-шлюза на портале Azure появляется следующее сообщение об ошибке: 

**Не удалось сохранить шлюз виртуальной сети &lt;имя шлюза&gt;. Недопустимое имя ресурса &lt;имя сертификата, который вы пытаетесь передать&gt;**.

### <a name="cause"></a>Причина:

Эта проблема возникает, когда имя сертификата содержит недопустимые знаки, например пробел. 

## <a name="azure-portal-error-vpn-package-file-download-error-503"></a>Ошибка портала Azure: ошибка 503 при скачивании файла пакета VPN

### <a name="symptom"></a>Симптом

При попытке скачать пакет конфигурации VPN-клиента появляется следующее сообщение об ошибке:

**Не удалось скачать файл. Сведения об ошибке: ошибка 503. Сервер занят.**
 
### <a name="solution"></a>Решение

Эта ошибка может быть вызвана временным сбоем сети. Попробуйте еще раз скачать пакет VPN через несколько минут.

## <a name="azure-vpn-gateway-upgrade-all-p2s-clients-are-unable-to-connect"></a>Обновление VPN-шлюза Azure, всем клиентам типа "точка — сеть" не удается подключиться

### <a name="cause"></a>Причина:

Если время существования сертификата составляет более 50 процентов, то сертификат сменяется.

### <a name="solution"></a>Решение

Чтобы устранить эту проблему, создайте и распространите новые сертификаты для VPN-клиентов. 

## <a name="too-many-vpn-clients-connected-at-once"></a>Слишком много одновременно подключенных VPN-клиентов

Для каждого VPN-шлюза максимальное число допустимых подключений равно 128.  Общее количество подключенных клиентов можно просмотреть на портале Azure.

## <a name="point-to-site-vpn-incorrectly-adds-a-route-for-100008-to-route-table"></a>VPN-подключение типа "точка — сеть" неправильно добавляет маршрут для 10.0.0.0/8 в таблицу маршрутов

### <a name="symptom"></a>Симптом

При установлении VPN-подключения с клиентом типа "точка — сеть" ожидается, что VPN-клиент добавит маршрут к виртуальной сети Azure, а служба Iphelper добавит маршрут для подсети VPN-клиентов. 

Тем не менее, если диапазон адресов VPN-клиента принадлежит подсети, меньшей 10.0.0.0/8, например 10.0.12.0/24, то вместо маршрута для 10.0.12.0/24 добавляется ошибочный маршрут для 10.0.0.0/8, который имеет более высокий приоритет. 

Это разрывает связь с другими локальными сетями, которые могут входить в другую подсеть в пределах диапазона 10.0.0.0/8, например 10.50.0.0/24, для которой не определен конкретный маршрут. 

### <a name="cause"></a>Причина:

Это сделано намеренно для клиентов Windows. При использовании протокола PPP IPCP клиент получает IP-адрес для интерфейса туннеля от сервера (в данном случае — VPN-шлюза). Однако из-за ограничений в протоколе у клиента нет маски подсети. Так как нет другого способа получить ее, клиент пытается угадать маску подсети на основе класса IP-адрес интерфейса туннеля. 

Следовательно, маршрут добавляется в зависимости от следующего статического сопоставления: 

если адрес относится к классу A, применяется маска /8;

если адрес относится к классу B, применяется маска /16;

если адрес относится к классу C, применяется маска /24.

##  <a name="vpn-client-cannot-access-network-file-shares"></a>VPN-клиент не может получить доступ к сетевым файловым ресурсам

### <a name="symptom"></a>Симптом

VPN-клиент подключился к сети Azure. Однако он не может получить доступ к сетевым папкам.

### <a name="cause"></a>Причина:

Для доступа к файловым ресурсам используется протокол SMB. Сбой происходит, когда VPN-клиент добавляет учетные данные сеанса при инициировании подключения. После установления подключения клиент принудительно использует кэшированные учетные данные для аутентификации Kerberos. Это инициирует отправку запросов к центру распространения ключей (контроллеру домена) для получения токена. Так как клиенты подключаются через Интернет, они могут не подключиться к контроллеру домена. Следовательно, клиенты не могут выполнить отработку отказа с Kerberos на NTLM. 

Единственный случай, в котором клиенту будет предложено указать учетные данные — когда у клиента есть действительный сертификат (в котором для параметра SAN указано имя участника-пользователя), выданный доменом, к которому присоединен клиент, и клиент физически подключен к доменной сети. В этом случае клиент пытается использовать сертификат и подключиться к контроллеру домена. KDC возвращает ошибку "KDC_ERR_C_PRINCIPAL_UNKNOWN".  Это вынуждает клиент выполнить отработку отказа на NTLM. 

### <a name="solution"></a>Решение

Чтобы обойти эту проблему, отключите кэширование учетных данных домена в следующем подразделе реестра: 

    HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Lsa\DisableDomainCreds - Set the value to 1 


## <a name="cannot-find-the-point-to-site-vpn-connection-in-windows-after-reinstalling-vpn-client"></a>Не удается найти VPN-подключение типа "точка — сеть" в Windows после повторной установки VPN-клиента

### <a name="symptom"></a>Симптом

Вы удаляете VPN-подключение типа "точка — сеть", а затем переустанавливаете VPN-клиент. В этом случае VPN-подключение не будет успешно настроено. Вы не видите VPN-подключение в параметрах **Сетевые подключения** в Windows.

### <a name="solution"></a>Решение

Чтобы устранить эту проблему, удалите старые файлы конфигурации VPN-клиента из папки **C:\Users\{имя_пользователя}\AppData\Roaming\Microsoft\Network\Connections**, а затем снова запустите установщик VPN-клиента.
