# <a name="overview-of-azure-devops-project"></a>Обзор службы проектов Azure DevOps

Служба проектов Azure DevOps упрощает начало работы с Azure. Ресурс проекта DevOps помогает запустить приложение избранного типа в службе Azure по своему выбору, просто выполнив несколько быстрых действий на портале Azure. Служба проектов DevOps устанавливает все компоненты, необходимые для разработки, развертывания и мониторинга приложения.
Панель мониторинга службы проектов DevOps позволяет отслеживать фиксацию кода, выполнение сборки и развертывания — и все это в одном представлении на портале Azure.

## <a name="why-should-i-use-the-azure-devops-project"></a>Почему следует использовать службу проектов Azure DevOps?

Служба проектов Azure DevOps автоматизирует установку всего конвейера непрерывной интеграции (CI) и непрерывной поставки (CD) в Azure.  Можно начать с существующего кода или использовать один из предоставленных примеров приложений и быстро развернуть это приложение в различных службах Azure, таких как "Виртуальные машины", "Служба приложений", "Служба контейнеров Azure", "База данных SQL Azure" и Azure Service Fabric.  

Служба проектов Azure DevOps выполняет всю работу по начальной настройке конвейера DevOps, включая настройку начального репозитория Git, настройку конвейера непрерывной интеграции и непрерывной поставки (CI/CD), создание ресурса Application Insights для мониторинга. Кроме того, она обеспечивает единое представление всего решения, создавая панель мониторинга службы проектов Azure DevOps на портале Azure.

С помощью службы проектов Azure DevOps можно:

* быстро развернуть приложение в Azure;
* автоматизировать настройку конвейера CI/CD в VSTS;
* использовать проект DevOps в качестве шаблона, чтобы понять, как правильно настроить непрерывную интеграцию и непрерывную доставку в Azure с помощью VSTS;
* приступить к работе с конвейером CI/CD в Azure и настроить конвейер выпуска в соответствии со своим сценарием.

## <a name="how-do-i-use-the-azure-devops-project"></a>Как использовать службу проектов Azure DevOps?

Служба проектов Azure DevOps доступна на портале Azure.  Вы можете создать ресурс проекта Azure DevOps на портале так же, как и любой другой ресурс Azure.  Служба проектов DevOps предоставляет пошаговые мастера для различных параметров конфигурации.  

В процессе начальной настройки можно выбрать несколько параметров конфигурации.  Вот какие параметры доступны:

* Использование предоставленного примера приложения или собственного кода.
* Выбор языка приложения.
* Выбор платформы приложения на основе языка.
* Выбор службы Azure (цель развертывания).
* Учетная запись VSTS (новая или существующая).
* Выбор подписки Azure.
* Выбор расположения служб Azure.
* Выбор разных ценовых категорий для служб Azure.

После использования службы проектов Azure DevOps можно будет удалить все ресурсы в одном месте — на панели мониторинга службы проектов Azure DevOps, доступной на портале Azure.

## <a name="azure-devops-project-and-vsts-integration"></a>Интеграция службы проектов Azure DevOps и VSTS

Проекты DevOps основаны на среде VSTS.  Служба проектов DevOps автоматизирует все действия, которые необходимо выполнить в VSTS для настройки непрерывной интеграции и непрерывной доставки в Azure.  Репозиторий Git создается в новой или существующей учетной записи VSTS.  Служба проектов DevOps фиксирует пример приложения или ваш код в новом репозитории Git.  Служба автоматизации также задает триггер непрерывной интеграции для сборки, поэтому после каждой фиксации нового кода инициируется выполнение сборки.  Кроме того, служба проектов DevOps создает триггер непрерывной доставки и развертывает все успешные новые сборки в службе Azure на ваш выбор.  Можно настроить определения сборки и выпуска для реализации дополнительных сценариев.  Можно также клонировать определения сборки и выпуска, чтобы использовать в других проектах.

После создания проекта DevOps можно:

* настраивать конвейер сборки и выпуска;
* отправлять запросы на вытягивание для управления потоком кода и обеспечения высокого уровня качества;
* тестировать все фиксируемые изменения и выполнять их сборку, прежде чем добавлять их в код, чтобы повысить качество;
* отслеживать невыполненную работу и проблемы проекта непосредственно в контексте приложения.
