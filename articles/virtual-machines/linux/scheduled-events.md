# <a name="azure-metadata-service-scheduled-events-preview-for-linux-vms"></a>Служба Azure метаданных: Запланированные события (Предварительная версия) для виртуальных машин Linux

> [!NOTE] 
> Предварительные версии предоставляются при условии, что вы соглашаетесь с условиями использования. Дополнительные сведения см. в статье [Дополнительные условия использования предварительных выпусков Microsoft Azure](https://azure.microsoft.com/support/legal/preview-supplemental-terms/).
>

Запланированные события является subservice в метаданных службы Azure, которая дает вашей времени приложения для подготовки для обслуживания виртуальной машины (VM). Предоставляет сведения о предстоящем обслуживании событий (например, перезагрузить), чтобы приложение можно подготовить для них и ограничить перебоев в работе. Он доступен для всех типов виртуальных машин Azure, включая PaaS и IaaS в Windows и Linux. 

Сведения о запланированных событиях, назначенные в Windows, см. в статье [Служба метаданных Azure. Запланированные события (предварительная версия) для виртуальных машин Windows](../windows/scheduled-events.md).

## <a name="why-use-scheduled-events"></a>Зачем использовать запланированные события?

Многие приложения могут выиграть от времени на подготовку для обслуживания виртуальной Машины. Время может быть использовано для выполнения задач конкретного приложения, которые повышают доступность, надежность и удобство обслуживания, включая: 

- Контрольная точка и восстановления.
- Сток подключений.
- Отработка отказа первичной реплики.
- Удаление из пула подсистемы балансировки нагрузки.
- Ведение журнала событий.
- Нормальное завершение работы.

С запланированные события приложения можно обнаружить при обслуживания будет возникать и запуск задач, чтобы ограничить его влияния.  

Запланированные события предусматривают события в следующих случаях:

- Инициированные платформы обслуживания (например, операционная система узла обновления)
- Пользователь запустил обслуживания (например, перезагрузки или повторном развертывании виртуальной Машины)

## <a name="the-basics"></a>Основные сведения  

  Метаданные службы предоставляет сведения о работающих виртуальных машин с помощью конечной точки REST, доступный из виртуальной Машины. Информация доступна через IP многосетевой, чтобы она не предоставляется вне виртуальной Машины.

### <a name="scope"></a>Область
Запланированные события доставляются:

- Все виртуальные машины в облачной службе.
- Все виртуальные машины в наборе доступности.
- Все виртуальные машины в группе размещения набора масштабирования. 

В результате проверки `Resources` в событие для определения, какие виртуальные машины затрагиваются.

### <a name="discover-the-endpoint"></a>Найти конечную точку
Для виртуальных машин, которые включены для виртуальных сетей является полной конечной точки для последней версии запланированные события: 

 > `http://169.254.169.254/metadata/scheduledevents?api-version=2017-08-01`

В тех случаях, где создается виртуальная машина в виртуальной сети, служба метаданных доступна из многосетевой статических IP-адресов, `169.254.169.254`.
Если не будет создана ВМ в виртуальной сети вариантов по умолчанию для облачных служб и классические ВМ для обнаружения IP-адрес для использования требуется дополнительная логика. Чтобы узнать, как [найти конечную точку узла](https://github.com/azure-samples/virtual-machines-python-scheduled-events-discover-endpoint-for-non-vnet-vm), см. в этом примере.

### <a name="versioning"></a>Управление версиями 
Запланированные события службы с версиями. Версии являются обязательными, и текущая версия — `2017-08-01`.

| Version (версия) | Заметки о выпуске | 
| - | - | 
| 2017-08-01 | <li> Удалено подчеркивание перед именами ресурсов для виртуальных машин Iaas.<br><li>Требование заголовок метаданных, принудительно применяется для всех запросов | 
| 2017-03-01 | <li>Общедоступной предварительной версии


> [!NOTE] 
> Предыдущие выпуски preview запланированные события, поддерживаемые {последние} в качестве api-version. Этот формат больше не поддерживается и в дальнейшем будет считаться устаревшим.

### <a name="use-headers"></a>Использование заголовков
При запросе метаданных службы, необходимо указать заголовок `Metadata:true` чтобы случайно не был перенаправлен запрос. Заголовок `Metadata:true` является обязательным для всех запросов запланированных событий. В результате сбоя для включения заголовка в запросе возникает ответ «Неправильный запрос» из метаданных службы.

### <a name="enable-scheduled-events"></a>Активация запланированных событий
При первом запуске запроса для запланированных событий Azure неявно включает функцию на виртуальной Машине. Поэтому ожидается, что задержки ответа на первый вызов до двух минут.

> [!NOTE]
> Запланированные события автоматически отключается для службы, если службе не вызывает конечную точку на один день. После отключения запланированные события для службы события не создаются для обслуживания, инициированной пользователем.

### <a name="user-initiated-maintenance"></a>Пользователь запустил обслуживания
Пользователь запустил обслуживания виртуальной Машины через портал Azure, API, CLI или PowerShell приводит запланированное событие. Вы затем можно проверить логику подготовки обслуживания в приложении, и можно подготовить приложение для обслуживания, инициированной пользователем.

При перезапуске виртуальной Машины, событие с типом `Reboot` запланирована. При повторном развертывании виртуальной Машины, событие с типом `Redeploy` запланирована.

> [!NOTE] 
> В настоящее время более 100 операций обслуживания, инициированной пользователем можно одновременно запланировать.

> [!NOTE] 
> В настоящее время обслуживания, инициированной пользователем, полученный в запланированные события не настраивается. Возможность настройки запланирована к следующему выпуску.

## <a name="use-the-api"></a>Использовать API-Интерфейс

### <a name="query-for-events"></a>Запрос сведений о событиях
Запланированные события можно запросить, выполнив следующий вызов:

#### <a name="bash"></a>Bash
```
curl -H Metadata:true http://169.254.169.254/metadata/scheduledevents?api-version=2017-08-01
```

Ответ содержит массив запланированных событий. Пустой массив означает, что в настоящее время события не запланированы.
В случае которых запланированные события, ответ содержит массив событий. 
```
{
    "DocumentIncarnation": {IncarnationID},
    "Events": [
        {
            "EventId": {eventID},
            "EventType": "Reboot" | "Redeploy" | "Freeze",
            "ResourceType": "VirtualMachine",
            "Resources": [{resourceName}],
            "EventStatus": "Scheduled" | "Started",
            "NotBefore": {timeInUTC},              
        }
    ]
}
```

### <a name="event-properties"></a>Свойства события
|Свойство  |  ОПИСАНИЕ |
| - | - |
| EventId | Глобальный уникальный идентификатор этого события. <br><br> Пример: <br><ul><li>602d9444-d2cd-49c7-8624-8643e7171297  |
| EventType | Влияние, которое оказывает это событие. <br><br> Значения: <br><ul><li> `Freeze`: ВМ запланирован для остановки на несколько секунд. ЦП приостанавливается, но она не влияет на память, открытых файлах или сетевых подключений. <li>`Reboot`: Перезагрузка запланирована виртуальной Машины. (Несохраненные памяти будет потеряна.) <li>`Redeploy`: Виртуальной Машины планируется переместить на другой узел. (Временных дисков будут потеряны.) |
| ResourceType | Тип ресурса, который влияет на это событие. <br><br> Значения: <ul><li>`VirtualMachine`|
| Ресурсы| Список ресурсов, которые затрагивает данное событие. Список будет гарантированно содержать машин из не более одного [домен обновления](manage-availability.md), но он может не содержать всех машин в UD. <br><br> Пример: <br><ul><li> ["FrontEnd_IN_0", "BackEnd_IN_0"] |
| Объект EventStatus | Состояние этого события. <br><br> Значения: <ul><li>`Scheduled`. Это запланированное событие состоится по истечении времени, указанного в свойстве `NotBefore`.<li>`Started`. Это событие запущено.</ul> Не `Completed` или когда-либо предоставляется как состояние. Это событие больше не возвращается после завершения события.
| NotBefore| Время, после чего можно запустить данное событие. <br><br> Пример: <br><ul><li> 2016-09-19T18:29:47Z  |

### <a name="event-scheduling"></a>Планирование события
Каждое событие запланирована минимальный объем времени в будущем в зависимости от типа события. Это время отражается в свойстве события `NotBefore`. 

|EventType  | Минимальная уведомления |
| - | - |
| Freeze| 15 минут |
| Reboot | 15 минут |
| Повторное развертывание | 10 минут |

### <a name="start-an-event"></a>События запуска 

После узнать предстоящего события и завершить свою логику для нормальное завершение работы, вы можете утвердить необработанных событий, делая `POST` вызвать службу метаданных с `EventId`. Указывает этот вызов в Azure, он может сократить уведомления минимальное время (если возможно). 

В следующем примере JSON в ожидается `POST` текст запроса. Запрос должен содержать список `StartRequests`. Каждый `StartRequest` содержит `EventId` требуется ускорить события:
```
{
    "StartRequests" : [
        {
            "EventId": {EventId}
        }
    ]
}
```

#### <a name="bash-sample"></a>Образец bash
```
curl -H Metadata:true -X POST -d '{"DocumentIncarnation":"5", "StartRequests": [{"EventId": "f020ba2e-3bc0-4c40-a10b-86575a9eabd5"}]}' http://169.254.169.254/metadata/scheduledevents?api-version=2017-08-01
```

> [!NOTE] 
> Подтверждение события позволяет событий для продолжения работы для всех `Resources` в событии, не только ВМ подтверждает события. Таким образом, вы можете выбрать заполнителем для координации подтверждение, которое может быть простым как первую машину в `Resources` поле.

## <a name="python-sample"></a>Пример на языке Python 

Следующий пример запрашивает метаданные службы запланированные события и утверждает каждый необработанных событий:

```python
#!/usr/bin/python

import json
import urllib2
import socket
import sys

metadata_url = "http://169.254.169.254/metadata/scheduledevents?api-version=2017-03-01"
headers = "{Metadata:true}"
this_host = socket.gethostname()

def get_scheduled_events():
   req = urllib2.Request(metadata_url)
   req.add_header('Metadata', 'true')
   resp = urllib2.urlopen(req)
   data = json.loads(resp.read())
   return data

def handle_scheduled_events(data):
    for evt in data['Events']:
        eventid = evt['EventId']
        status = evt['EventStatus']
        resources = evt['Resources']
        eventtype = evt['EventType']
        resourcetype = evt['ResourceType']
        notbefore = evt['NotBefore'].replace(" ","_")
        if this_host in resources:
            print "+ Scheduled Event. This host is scheduled for " + eventype + " not before " + notbefore
            # Add logic for handling events here

def main():
   data = get_scheduled_events()
   handle_scheduled_events(data)
   
if __name__ == '__main__':
  main()
  sys.exit(0)
```
