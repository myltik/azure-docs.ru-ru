# <a name="azure-active-directory-b2b-collaboration-invitation-redemption"></a>Активация приглашения службы совместной работы Azure Active Directory B2B

При взаимодействии с партнерскими организациями в службе совместной работы Azure Active Directory (Azure AD) B2B вы можете пригласить гостевых пользователей для доступа к общим приложениям. После добавления в каталог с помощью пользовательского интерфейса или приглашения с помощью PowerShell гостевой пользователь должен получить первоначальное согласие, приняв [условия конфиденциальности информации](#privacy-policy-agreement). Для этого есть несколько способов:

- Пользователь, приглашающий гостя, отправляет прямую ссылку в общее приложение. Приглашенный открывает ссылку для входа, принимает условия конфиденциальности информации и беспрепятственно получает доступ к общему ресурсу (гостевой пользователь все равно получает электронное письмо-приглашение с URL-адресом активации, но ему больше не требуется использовать приглашение по электронной почте, за исключением особых случаев).  
- Гостевой пользователь получает приглашение по электронной почте и открывает щелчком URL-адрес активации. При первом входе в систему пользователю предлагается принять условия конфиденциальности информации.

## <a name="redemption-through-a-direct-link"></a>Активация с помощью прямой ссылки

Можно пригласить гостевого пользователя, выслав ему прямую ссылку на общее приложение. Для гостевого пользователя процесс активации так же прост, как вход в приложение, к которому ему предоставили доступ. Пользователь может щелкнуть ссылку в приложении, просмотреть и принять условия конфиденциальности информации, а затем беспрепятственно получить доступ к приложению. В большинстве случаев гостевым пользователям больше не нужно открывать щелчком URL-адрес активации в приглашении по электронной почте.

Если вы пригласили гостевых пользователей с помощью пользовательского интерфейса или решили отправить приглашение по электронной почте с помощью PowerShell, приглашенные пользователи все равно получат приглашение по электронной почте. Это сообщение полезно в следующих особых случаях:

- У пользователя нет учетной записи Azure AD или учетной записи Майкрософт (MSA). В этом случае пользователю необходимо создать MSA до перехода по ссылке. Или он может использовать URL-адрес активации в приглашении по электронной почте. В процессе активации пользователю автоматически отправляется запрос на создание MSA.
- Иногда объект приглашенного пользователя не может использовать адрес электронной почты из-за конфликта с контактным объектом (например, контактным объектом Outlook). В этом случае пользователю необходимо щелкнуть URL-адрес активации в приглашении, полученном по электронной почте.
- Пользователь может выполнить вход с псевдонимом адреса электронной почты, по которому поступило приглашение (псевдоним — это дополнительный адрес электронной почты, связанный с учетной записью электронной почты). В этом случае пользователю необходимо щелкнуть URL-адрес активации в приглашении, полученном по электронной почте.

Если эти особые случаи актуальны для организации, рекомендуем приглашать пользователей с отправкой приглашения по электронной почте. Кроме того, если ситуация не описана ни в одном из упомянутых особых случаев, пользователь по-прежнему может щелкнуть URL-адрес в приглашении, полученном по электронной почте, чтобы получить доступ.

## <a name="redemption-through-the-invitation-email"></a>Активация с помощью приглашения по электронной почте

При приглашении по электронной почте пользователи также могут активировать приглашение из этого сообщения. Приглашенный пользователь может щелкнуть URL-адрес активации в сообщении электронной почты, а затем просмотреть и принять условия конфиденциальности информации. Более подробно процесс описан здесь:

1.  Приглашенный пользователь получает сообщение электронной почты от **службы приглашений Майкрософт**.
2.  Приглашенный пользователь выбирает **Начать работу** в сообщении электронной почты.
3.  Если у приглашенного пользователя нет учетной записи Azure AD или MSA, ему будет предложено создать MSA.
4.  Приглашенный пользователь перенаправляется к экрану **просмотра разрешений**, где может просмотреть заявление о конфиденциальности приглашающей организации и принять условия.

## <a name="privacy-policy-agreement"></a>Соглашение о политике конфиденциальности

После первого входа любого гостевого пользователя для доступа к ресурсам партнерской организации для него отображается экран **Review permissions** (Просмотр разрешений). Здесь он может просмотреть заявление о конфиденциальности приглашающей организации. Чтобы продолжить, пользователь должен принять условия использования своих данных в соответствии с политиками конфиденциальности приглашающей организации.

![Снимок экрана с настройками пользователя на панели доступа](media/redemption-experience/ConsentScreen.png) 

Сведения о том, как администратор клиента может получить ссылку на заявление о конфиденциальности организации, см. в статье [How-to: Add your organization's privacy info in Azure Active Directory](https://aka.ms/adprivacystatement) (Практическое руководство. Добавление сведений о конфиденциальности организации в Azure Active Directory).
