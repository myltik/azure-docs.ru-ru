# <a name="roll-out-password-reset-for-users"></a>Развертывание функции сброса паролей для пользователей

Большинство клиентов выполняют следующие шаги для обеспечения плавного развертывания функции самостоятельного сброса пароля (SSPR).

1. [Включите сброс паролей в каталоге.](active-directory-passwords-getting-started.md)
2. [Настройте локальные разрешения AD для обратной записи паролей.](active-directory-passwords-how-it-works.md#active-directory-permissions)
3. [Настройте обратную запись паролей](active-directory-passwords-writeback.md#configuring-password-writeback) для записи паролей Azure AD обратно в ваш локальный каталог.
4. [Назначьте и проверьте необходимые лицензии.](active-directory-passwords-licensing.md)
5. Если вы хотите выполнить развертывание постепенно, можно разрешить сброс паролей только для группы пользователей. Для этого переключите параметр **Разрешен самостоятельный сброс пароля** со значения **Все** на **Группа** и выберите группу безопасности, для которой будет разрешен сброс паролей. Членам этой группы должны быть назначены лицензии. Это отличный способ использовать [групповое лицензирование](active-directory-passwords-licensing.md#enable-group-or-user-based-licensing).
6. На основе вашей политики заполните минимальный набор [данных проверки подлинности](active-directory-passwords-data.md).
7. Отправьте пользователям инструкции по регистрации и сбросу пароля с помощью SSPR.
    > [!NOTE]
    > Тестируйте SSPR с использованием учетной записи пользователя, а не администратора, так как Майкрософт имеет строгие требования к проверке подлинности учетных записей Azure типа "администратор". Дополнительные сведения о политике паролей администратора см. в статье [Как работает управление паролями в Azure Active Directory](active-directory-passwords-how-it-works.md).

8. Вы можете инициировать регистрацию в любой момент и потребовать от пользователей подтвердить свои данные аутентификации через определенный промежуток времени. Если вы не хотите, чтобы ваши пользователи регистрировались, вы можете [развернуть функцию сброса пароля без необходимости регистрации конечного пользователя](active-directory-passwords-data.md).
9. Через некоторый промежуток времени просмотрите сведения о зарегистрированных пользователях и об использовании ими этой функции в [отчетах, предоставленных Azure AD](active-directory-passwords-reporting.md).

## <a name="email-based-rollout"></a>Развертывание по электронной почте

Многие клиенты считают рассылку простых инструкций по почте самым легким способом убедить пользователей использовать SSPR. Мы создали три простых письма, которые можно использовать в качестве шаблонов для упрощения процесса развертывания.

* **Скоро будет доступен** шаблон сообщения электронной почты, которое отправляется за несколько недель или дней перед развертыванием, чтобы пользователи знали, что им нужно что-то сделать.
* **Уже доступен** шаблон сообщения электронной почты, который отправляется в день запуска, чтобы пользователи зарегистрировались и подтвердили свои данные проверки подлинности и могли использовать SSPR, когда это необходимо.
* Шаблон сообщения электронной почты с **напоминанием о регистрации**, который отправляется спустя несколько дней или недель после развертывания, чтобы напомнить пользователям о необходимости регистрации и подтверждения данных для проверки подлинности.

## <a name="creating-your-own-password-portal"></a>Создание собственного портала паролей

Многие из наших крупных клиентов предпочитают размещать веб-страницы и создавать корневые DNS-записи, например https://passwords.contoso.com. Они заполняют эту страницу ссылками на сброс пароля Azure AD, регистрацию сброса пароля, порталы изменения пароля и другие сведения для конкретной организации. Вы можете включить во все сообщения электронной почты и рекламные листовки один запоминающийся URL-адрес, с помощью которого пользователи смогут начать работу со службой.

* Портал сброса пароля: https://passwordreset.microsoftonline.com/
* Портал регистрации для сброса пароля: http://aka.ms/ssprsetup
* Портал изменения пароля: https://account.activedirectory.windowsazure.com/ChangePassword.aspx

## <a name="using-enforced-registration"></a>Принудительная регистрация

Если нужно, чтобы пользователи зарегистрировались для сброса пароля, вы можете принудительно регистрировать их при входе с помощью Azure AD. Эту функцию можно включить в колонке **Сброс паролей** для своего каталога, активировав параметр **Требовать регистрацию пользователей при входе?** на вкладке **Регистрация**.

Администраторы могут потребовать от пользователей повторной регистрации по истечении определенного периода времени, установив **количество дней, по истечении которых пользователям будет предложено повторно подтвердить данные проверки подлинности** в промежутке 0–730 дней.

После того как вы включите этот параметр, регистрирующиеся пользователи увидят сообщение о том, что их администратор запрашивает у них подтверждение данных проверки подлинности.

## <a name="populate-authentication-data"></a>Заполнение данных для проверки подлинности

Если вы [заполните данные аутентификации своих пользователей](active-directory-passwords-data.md), им не нужно будет регистрироваться для сброса пароля, чтобы использовать SSPR. Пользователи могут сбрасывать свои пароли, если их данные проверки подлинности соответствуют политике сброса пароля, которую вы определили.

## <a name="disabling-self-service-password-reset"></a>Выключение самостоятельного сброса пароля

Отключить самостоятельный сброс пароля так же просто, как и включить. Откройте ваш клиент Azure AD, щелкните **Сброс пароля**, **Свойства**, и выберите **Никто** для параметра **Разрешен самостоятельный сброс пароля**.
