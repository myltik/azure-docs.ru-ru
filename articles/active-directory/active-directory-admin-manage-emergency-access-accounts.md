# <a name="manage-emergency-access-administrative-accounts-in-azure-ad"></a>Управление учетными записями администратора для аварийного доступа в Azure AD 

Для большинства повседневных действий пользователям не нужны права *глобального администратора*. Пользователям не следует бессрочно назначать роль, потому что они могут непреднамеренно выполнить задачу, требующую более высоких разрешений, чем у них должно быть. Когда пользователям не нужно выступать в качестве глобального администратора, им нужно активировать назначение роли с помощью Azure Active Directory (Azure AD) Privileged Identity Management (PIM) в собственной учетной записи или в альтернативной учетной записи администратора.

Кроме ограничения прав доступа администратора для пользователей необходимо предотвратить непреднамеренное блокирование администрирования вашего клиента Azure AD, так как в этом случае вы не сможете войти или активировать учетную запись отдельного пользователя в качестве администратора. Уменьшить ущерб от случайного блокирования административного доступа можно, сохранив в клиенте несколько *учетных записей для аварийного доступа*.

Учетные записи аварийного доступа могут помочь организациям ограничить привилегированный доступ в имеющейся среде Azure Active Directory. Такие учетные записи имеют высокий уровень привилегий и не присвоены конкретным лицам. Учетные записи для аварийного доступа ограничены непредвиденными ситуациями, в которых невозможно использовать обычные учетные записи администратора. Организации должны ограничивать использования учетной записи для аварийного доступа только при необходимости.

Организации может потребоваться использовать учетные записи для аварийного доступа в следующих ситуациях.

 - Учетные записи пользователей объединяются в федерацию, которая в настоящее время недоступна из-за перебоев в работе сотовой сети или поставщика удостоверений. Например, пользователи, возможно, не смогут выполнить вход при перенаправлении к поставщику удостоверений в Azure AD, если узел поставщика удостоверений в среде перестал работать. 
 - Администраторы зарегистрированы с помощью службы "Многофакторная идентификация Azure", а все их устройства недоступны. Пользователям, возможно, не удается завершить многофакторную проверку подлинности для активации роли. Например, из-за сбоя сотовой сети они не могут отвечать на телефонные звонки или получать текстовые сообщения (два механизма аутентификации, зарегистрированные для устройства). 
 - Пользователь, которому недавно предоставили права глобального административного доступа, покинул организацию. Azure AD предотвращает удаление последней учетной записи *глобального администратора*, но не мешает удалению или отключению локальной учетной записи. В этих случаях сбой может привести к тому, что организация не сможет восстановить учетную запись.

## <a name="initial-configuration"></a>Исходная конфигурация

Создайте две или больше учетных записей для аварийного доступа. Они должны быть облачными учетными записями, использующими домен \*.onmicrosoft.com, а также не объединенными в федерации или не синхронизированными из локальной среды. 

Учетные записи не должны быть связаны с отдельным пользователем в организации. Организации должны обеспечить защиту учетных данных для этих учетных записей и предоставить к ним доступ только тем, у кого есть право на их использование. 

> [!NOTE]
> Пароль учетной записи для экстренного доступа обычно разделяется на две или три части, написанные на отдельных листах бумаги, хранящихся в безопасных огнестойких сейфах в отдельных безопасных местах. 
>
> Убедитесь, что ваши учетные записи аварийного доступа не связаны ни с мобильными телефонами, предоставленными сотрудниками, ни с аппаратными маркерами отдельных сотрудников, ни с другими учетными данными конкретных сотрудников. Эта мера безопасности распространяется на случаи, когда учетные данные нужны, а отдельный сотрудник недоступен. 

### <a name="initial-configuration-with-permanent-assignments"></a>Исходная конфигурация с постоянным присвоением

Один из вариантов — назначить пользователям роль *глобального администратора* на постоянной основе. Этот вариант будет уместен для организаций без подписок Azure AD Premium P2.

Чтобы снизить риск атаки в результате скомпрометированного пароля, Azure AD рекомендует требовать многофакторную поверку подлинности всех отдельных пользователей. Эта группа должна включать администраторов и остальных специалистов (например, финансовых сотрудников), чья скомпрометированная учетная запись окажет значительное влияние. 

Если в организации устройства не используются совместно, то для этих учетных записей для аварийного доступа может быть невозможна многофакторная проверка подлинности. При настройке политики условного доступа для требования [регистрации путем многофакторной проверки подлинности для каждого администратора](https://docs.microsoft.com/azure/multi-factor-authentication/multi-factor-authentication-get-started-user-states) для Azure AD и других приложений на основе модели "программное обеспечение как услуга" (SaaS) может понадобиться настроить исключения политики, чтобы исключить учетные записи для аварийного доступа из этого требования.

### <a name="initial-configuration-with-approvals"></a>Исходная конфигурация с утверждениями

Другой вариант — это настроить пользователей как правомочных и утверждающих для активации роли *глобального администратора*. Этот вариант требует от вашей организации наличие подписок Azure AD Premium P2. Для этого также потребуется вариант многофакторной проверки подлинности, подходящий для совместного использования несколькими людьми и для сетевой среды. Эти требования обусловлены тем, что для активации роли *глобального администратора* пользователи сначала должны выполнить многофакторную проверку подлинности. Дополнительные сведения см. в разделе [How to require MFA in Azure AD Privileged Identity Management](https://docs.microsoft.com/azure/active-directory/active-directory-privileged-identity-management-how-to-require-mfa) (Требование MFA в Azure AD Privileged Identity Management).

Не рекомендуется использовать многофакторную проверку подлинности, связанную с персональными устройствами для учетных записей для аварийного доступа. В случае чрезвычайной ситуации может оказаться, что пользователь, которому необходим доступ к устройству, зарегистрированному через многофакторную проверку подлинности, сам не имеет личного устройства. 

Кроме того, следует учитывать характер угроз. Например, может возникнуть непредвиденное обстоятельство, такое как чрезвычайная ситуация в случае стихийного бедствия, при котором мобильный телефон или другие сети могут быть недоступны. Важно обеспечить, чтобы любые зарегистрированные устройства хранились в известном безопасном месте с несколькими средствами связи с Azure AD.

## <a name="ongoing-monitoring"></a>Постоянный мониторинг

Отслеживайте [журналы входа и аудита Azure AD](https://docs.microsoft.com/azure/active-directory/active-directory-reporting-activity-sign-ins) при выполнении любых действий, связанных с входом и аудитом, из учетных записей для аварийного доступа. Обычно из этих учетных записей не предполагается входить и вносить изменения, поэтому их использование, вероятно, будет аномальным и потребует расследования службы безопасности.

## <a name="account-check-validation-must-occur-at-regular-intervals"></a>Регулярные проверки учетной записи

Чтобы проверить учетную запись, выполняйте следующие шаги в таких случаях:
- Каждые 90 дней.
- Когда произошли недавние изменения в составе ИТ-персонала, такие как смена работы, увольнение или наем нового сотрудника.
- При изменении подписок Azure AD в организации.

Для обучения сотрудников использованию учетных записей для аварийного доступа сделайте следующее.

* Убедитесь, что сотрудники службы мониторинга безопасности знают, что действия по проверке учетных записей нужно выполнять регулярно.
* Убедитесь, что в облачных учетных записях пользователей можно выполнить вход и активировать роли и что пользователи, которым может потребоваться сделать это во время чрезвычайной ситуации, проходят обучение по этому процессу.
* Убедитесь, что в любом устройстве отдельного пользователя или личных данных не зарегистрирована многофакторная проверка подлинности или самостоятельный сброс пароля (SSPR). 
* Если учетные записи зарегистрированы для многофакторной проверки подлинности на устройстве для использования во время активации роли, убедитесь, что устройство доступно для всех администраторов, которым может потребоваться использовать его в случае чрезвычайной ситуации. Кроме того, убедитесь, что устройство зарегистрировано с по меньшей мере двумя механизмами с разными режимами сбоя. Например, устройство может осуществлять связь с Интернетом через беспроводную сеть объекта и через сеть сотового оператора.
* Обновите учетные данные учетной записи.
