# <a name="conditional-access-in-azure-active-directory"></a>Условный доступ в Azure Active Directory

Безопасность очень важна для организаций, использующих облако. Ключевым аспектом облачной безопасности является идентификация и доступ при управлении облачными ресурсами. В мире мобильных и облачных технологий пользователи могут откуда угодно обращаться к ресурсам организации с помощью различных устройств и приложений. Поэтому уже недостаточно просто указать, кто имеет доступ к ресурсу. Чтобы управлять балансом между безопасностью и производительностью, в решении по управлению доступом ИТ-специалисты также должны учитывать способ доступа к ресурсам. Это требование можно выполнить с помощью условного доступа Azure AD. Условный доступ — это функция Azure Active Directory, которая позволяет принудительно применять элементы управления доступом к приложениям в среде с учетом определенных условий из центрального расположения. 


![Контроль](./media/active-directory-conditional-access-azure-portal/81.png)

В этой статье приведены общие сведения об условном доступе в Azure AD.


## <a name="common-scenarios"></a>Распространенные сценарии

В эпоху мобильных и облачных технологий Azure Active Directory обеспечивает единый вход для устройств, приложений и служб из любого расположения. В связи с увеличением количества устройств (включая BYOD) и сторонних приложений SaaS, а также растущей популярностью работы за пределами корпоративных сетей ИТ-специалисты все чаще сталкиваются с двумя противоположными задачами:

- продуктивная работа пользователей в любом месте и в любое время;
- постоянная защита корпоративных ресурсов.

С помощью политик условного доступа можно применить элементы управления доступом в соответствии с необходимыми условиями. Условный доступ Azure AD позволяет при необходимости повысить безопасность и не влияет на работу пользователей, когда это не требуется. 

Ниже приведено несколько распространенных проблем доступа, которые может устранить условный доступ.



- **[Риск при входе](active-directory-conditional-access-conditions.md#sign-in-risk)**. Служба "Защита идентификации Azure Active Directory" обнаруживает риски при входе. Как ограничить доступ, если обнаруженный риск при входе указывает на несоответствующий субъект? Что, если вы хотите получить более весомое свидетельство того, что вход был выполнен правомочным пользователем, или ваши сомнения достаточно сильны, чтобы заблокировать определенным пользователям доступ к приложению?

- **[Расположение в сети](active-directory-conditional-access-locations.md)**. Служба Azure AD доступна откуда угодно. Что, если выполняется попытка доступа из расположения в сети, которое не находится под управлением вашего ИТ-отдела? Использование сочетания имени пользователя и пароля может быть достаточным для подтверждения подлинности при доступе к ресурсам из корпоративной сети. Что, если требуется более надежное подтверждение подлинности при попытках доступа, инициируемых из других непредвиденных стран или регионов мира? Что, если необходимо даже блокировать попытки доступа из определенных расположений?  

- **[Управление устройствами](active-directory-conditional-access-conditions.md#device-platforms)**. В Azure AD пользователи могут обращаться к облачным приложениям, используя широкий спектр устройств, включая мобильные, а также используя личные устройства. Что, если эти попытки доступа должны будут предприниматься только с устройств, управляемых вашим ИТ-отделом? Что, если требуется даже блокировать доступ с устройств определенных типов к облачным приложениям в вашей среде? 

- **[Клиентское приложение](active-directory-conditional-access-conditions.md#client-apps)**. Сейчас для доступа ко многим облачным приложениям используются разные типы приложений, в том числе веб-приложения, мобильные приложения или классические приложения. Что, если попытка доступа выполняется с использованием типа клиентского приложения, вызывающего известные проблемы? Что, если для определенных типов приложений требуется устройство, которым управляет ваш ИТ-отдел? 

Эти вопросы и ответы на них дают представление о распространенных сценариях доступа, в которых применяется условный доступ Azure AD. Условный доступ — это компонент Azure Active Directory, который позволяет обрабатывать сценарии доступа, используя подход на основе политик.


## <a name="conditional-access-policies"></a>Политики условного доступа

Политика условного доступа — это определение сценария доступа с использованием следующего шаблона.

![Контроль](./media/active-directory-conditional-access-azure-portal/10.png)

Элемент **Then do this** (Сделать это) определяет ответ политики. Важно отметить, что цель политики условного доступа — не предоставление доступа к облачному приложению. В Azure AD для предоставления доступа к облачным приложениям используется назначение пользователей. С помощью политики условного доступа можно управлять способом доступа авторизованных пользователей (пользователей, которым предоставлен доступ к облачному приложению) к облачным приложениям при определенных условиях. С помощью ответа можно принудительно применить дополнительные требования, например прохождение многофакторной проверки подлинности, использование управляемого устройства и т. д. В контексте условного доступа Azure AD требования, которые принудительно применяет политика, называются элементами управления доступом. Наиболее строгая форма политики может блокировать доступ. Дополнительные сведения см. в разделе [Элементы управления условным доступом в Azure Active Directory](active-directory-conditional-access-controls.md).
     

Элемент **When this happens** (В этом случае) определяет причину активации политики. Эта причина определяется группой условий, которые должны быть соблюдены. В условном доступе Azure AD особую роль играют два условия назначения.

- **[Пользователи](active-directory-conditional-access-conditions.md#users-and-groups)**. Пользователи, выполняющие попытку доступа (**кто**). 

- **[Облачные приложения](active-directory-conditional-access-conditions.md#cloud-apps)**. Цели для попытки доступа (**что**).    

Эти два условия являются обязательными в политике условного доступа. Помимо этих двух обязательных условий можно также добавить дополнительные условия, которые описывают, как выполняется попытка доступа. К распространенным примерам можно отнести использование мобильных устройств или расположений, которые находятся за пределами корпоративной сети. Дополнительные сведения см. в разделе [Условия в условном доступе Azure Active Directory](active-directory-conditional-access-conditions.md).   

Сочетание условий с элементами управления доступом и представляет собой политику условного доступа. 

![Контроль](./media/active-directory-conditional-access-azure-portal/51.png)

С помощью условного доступа Azure AD можно контролировать доступ авторизованных пользователей к облачным приложениям. Цель политики условного доступа — применение дополнительных элементов управления доступом при попытке доступа к облачному приложению на основе способа, которым осуществляется эта попытка.

Одно из преимуществ подхода на основе политик для защиты доступа к облачным приложениям заключается в том, что вы можете начать проектировать требования политики для своей среды, используя структуру, описанную в этой статье, и не беспокоясь о технической реализации. 

## <a name="what-you-need-to-know"></a>Это важно знать

### <a name="general-requirements-for-using-conditional-access"></a>Общие требования для использования условного доступа

Условный доступ Azure AD можно использовать для защиты облачных приложений при поступлении запроса на аутентификацию из:

- веб-браузера;

- клиентского приложения, использующего современную аутентификацию;

- Exchange ActiveSync

Дополнительные сведения см. в разделе [Клиентские приложения](active-directory-conditional-access-conditions.md#client-apps).

Некоторые [облачные приложения](active-directory-conditional-access-conditions.md#cloud-apps) также поддерживают устаревшие протоколы аутентификации. Это относится, например, к SharePoint Online и Exchange Online. Когда клиентское приложение может использовать устаревший протокол аутентификации для доступа к облачному приложению, Azure AD не может принудительно применить политику условного доступа к этой попытке получения доступа. Чтобы предотвратить обход применяемых политик клиентским приложением, следует проверить, можно ли включить для затронутых облачных приложений только современную аутентификацию.

Ниже приведены примеры клиентских приложений, не подходящих для условного доступа:

- Office 2010 и более ранних версий;

- Office 2013, если не включена современная аутентификация.

Дополнительные сведения см. в статье [Настройка SharePoint Online и Exchange Online для условного доступа Azure Active Directory](active-directory-conditional-access-no-modern-authentication.md).


### <a name="license-requirements-for-using-conditional-access"></a>Требования лицензий для использования условного доступа

Для использования условного доступа требуется лицензия Azure AD Premium. Чтобы найти подходящую лицензию, ознакомьтесь с разделом [Сравнение общедоступных функций выпусков Free, Basic и Premium](https://www.microsoft.com/en-us/cloud-platform/azure-active-directory-features).
